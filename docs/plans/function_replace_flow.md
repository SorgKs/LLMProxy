# Полный поток замены function_replace → apply_diff (с отправкой read_file клиенту)

## 1. Начальный запрос (клиент → прокси)
- Запрос содержит ответ LLM с `function_replace` (path, function, code).
- Содержимого целевого файла в запросе **нет**.

## 2. Первичный анализ в прокси
- `AnswerProcessor.process_single_tool_call` извлекает параметры `function_replace`.
- Вызывает `_extract_file_content_from_request` — контент не найден.
- Результат: файл недоступен, конвертировать пока нельзя.

## 3. Решение: запрашиваем файл через read_file (отправляем клиенту)
- Прокси сохраняет оригинальный вызов `function_replace` во внутреннюю структуру (`self._pending_function_replace`).
- Формирует **минимальный ответ для клиента**, содержащий только:
  ```json
  {
    "id": "gen-xyz",
    "type": "function",
    "function": {
      "name": "read_file",
      "arguments": "{\"path\": \"<path>\", \"mode\": \"slice\", \"offset\": 1}"
    }
  }
  ```
- Никаких других tool_calls (включая исходное `function_replace`) в ответе нет.

## 4. Действие клиента
- Клиент получает `read_file`-ответ, читает файл локально и отправляет новый запрос в прокси с результатом `read_file`.

## 5. Второй проход прокси — локальная обработка (почему не отправляем LLM)
- Новый запрос содержит результат `read_file` (содержимое файла).
- Прокси видит отложенный `function_replace` (из п.3).
- Для `function_replace` уже есть всё необходимое (путь, имя, новый код, текущее содержимое файла), поэтому **нет необходимости запрашивать LLM**.
- Процесс продолжается локально.
- `_extract_file_content_from_request` собирает известные строки файла.
- `_detect_full_function_presence` проверяет, присутствует ли полный код функции (has_full_code).
- Если да — строится `apply_diff`:
  - извлекается старый код (строки start..end),
  - формируется diff с `<<<<<<< SEARCH`, `:start_line:`, `-------`, `=======`, `>>>>>>> REPLACE`.
- Создаётся tool_call `apply_diff` и помещается в ответ.

## 6. Финальный ответ клиенту (прокси → клиент)
- Ответ содержит только `apply_diff`.
- Клиент применяет diff локально.

## 7. Итог
- `function_replace` **никогда не попадает к клиенту** — остаётся внутри прокси.
- При отсутствии файла прокси отправляет минимальный `read_file`-ответ.
- После получения файла прокси сам формирует `apply_diff` без участия LLM.

## 8. Вспомогательные функции (уже спланированы)
- `_extract_file_content_from_request` — сбор известных строк файла из текущих данных.
- `_detect_full_function_presence` — проверка полноты кода (с комментарием перед функцией).

## 9. Настройки и файлы
- `config/tools.yaml` не меняется (function_replace остаётся включённым).
- `tools/function_replace.json` остаётся как есть (нужно для описания возможности LLM).
