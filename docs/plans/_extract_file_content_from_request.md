# План реализации функции `_extract_file_content_from_request`

## Цель
Собрать максимально полное представление о содержимом файла, используя все `read_file`‑tool‑calls, которые уже присутствуют в запросе/ответе от клиента. Функция будет использоваться перед конвертацией `function_replace` в `apply_diff`.

## Входные параметры
* **request_body** – словарь, полученный из HTTP‑запроса клиента. Содержит массив `messages`, в каждом из которых могут быть `tool_calls`.
* **target_path** – строка, путь к файлу, для которого нужно собрать содержимое.

## Алгоритм (словесное описание)

1. **Инициализация**
   - Создаём пустой словарь `file_lines` (ключ – номер строки (1‑based), значение – текст строки).

2. **Обход сообщений**
   - Проходим по массиву `request_body["messages"]` **в обратном порядке** (от последнего сообщения к первому). Это гарантирует, что более новые куски файла переопределяют старые.
   - В каждом сообщении проверяем наличие поля `tool_calls`.

3. **Фильтрация `read_file`‑calls**
   - Для каждого `tool_call` проверяем:
     * `function.name` == `"read_file"`
     * Параметр `path` в `arguments` точно совпадает с `target_path`.
   - Все остальные игнорируем.

4. **Разбор аргументов**
   - Десериализуем JSON‑строку `arguments`.
   - Получаем два ключевых поля:
     * `offset` – номер строки, с которой начинается возвращённый кусок (1‑based).
     * `content` – текст, который клиент вернул. Может быть одной строкой (с `\n`) или массивом строк.

5. **Нормализация `content`**
   - Если `content` – одна строка, разбиваем её по `\n` → список строк.
   - Если уже массив – используем как есть.
   - Убираем возможные пустые строки в конце.

6. **Заполнение `file_lines`**
   - Для каждой строки из полученного списка вычисляем её реальный номер в файле:
     `line_no = offset + index_in_chunk` (где `index_in_chunk` начинается с 0).
   - Если `line_no` уже присутствует в `file_lines`, **не перезаписываем** (потому что более новый кусок уже записан).
   - Иначе сохраняем `file_lines[line_no] = строка`.

7. **Сборка полного списка строк**
   - Находим максимальный номер строки `max_line = max(file_lines.keys(), default=0)`.
   - Создаём список `full_content` длиной `max_line`, заполненный пустыми строками.
   - Заполняем его из `file_lines`: `full_content[line_no‑1] = text`.

8. **Обработка «дыр»**
   - Если после заполнения в `full_content` остались пустые элементы, это означает, что часть файла ещё не была получена (например, файл читался кусками, и некоторые участки пропущены). Такие «пустые» строки остаются пустыми – при попытке построить diff они будут считаться неизвестными.

9. **Возврат**
   - Функция возвращает список `full_content`. Индекс = номер строки ‑ 1, т.е. `full_content[0]` – первая строка файла.

## Как будет использоваться
* В `AnswerProcessor.process_single_tool_call` перед попыткой конвертации `function_replace` вызываем `_extract_file_content_from_request`.
* Если полученный список полностью пустой, генерируем `read_file`‑tool‑call, откладываем текущий `function_replace` и ждём новый запрос от клиента с содержимым файла.
* Если список содержит хотя бы одну строку, передаём его в метод, который ищет нужную функцию и формирует `apply_diff`.

## Примечания
* Обход сообщений **от конца к началу** критичен – более новые ответы переопределяют старые.
* Функция работает только с `read_file`; любые другие инструменты игнорируются.
* При частичном чтении (разные `offset` и разные куски) функция «склеивает» их в один целостный массив строк.
